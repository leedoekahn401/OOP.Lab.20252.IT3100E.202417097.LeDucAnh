Ans: When iterating a List<Media> and calling toString() on each element, dynamic dispatch (polymorphism) ensures that each object's own toString() is called — Book prints "Book - title - category : cost $", DVD prints "DVD - title - category - director - length : cost $", CD prints "CompactDisc - title - category : cost $". Suppose we are taking this Comparable interface approach.
- What class should implement the Comparable interface?
Ans: The abstract Media class should implement Comparable<Media>, since all media types (Book, DVD, CD) are stored in a List<Media> and sorted together.

- In those classes, how should you implement the compareTo() method be to reflect the ordering that we want?
Ans: compareTo() should compare by title first (using String.compareTo()), then by cost (using Double.compare()), so items are naturally ordered by title, with cost as a tiebreaker.

- Can we have two ordering rules of the item (by title then cost and by cost then title) if we use this Comparable interface approach?
Ans: No. Comparable allows only one natural ordering via a single compareTo() method. For multiple ordering rules, use separate Comparator implementations instead.

- Suppose the DVDs has a different ordering rule from the other media types, that is by title, then decreasing length, then cost. How would you modify your code to allow this?
Ans: Overriding compareTo() in DigitalVideoDisc is not reliable because it would be called polymorphically on a List<Media>, causing inconsistency when comparing DVD against Book/CD. Instead, create a custom Comparator (e.g., DvdComparatorByTitleLengthCost) that checks instanceof and applies the DVD-specific rule, then pass it to Collections.sort() when needed.
